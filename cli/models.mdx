Manage the model registry used to resolve model names across different agents.

## Usage

```bash
harness models <subcommand>
```

## Subcommands

### list

List all models in the registry.

```bash
harness models list [OPTIONS]
```

**Options:**

<ParamField path="--json" type="boolean" default="false">
  Output as JSON instead of human-readable text
</ParamField>

<ParamField path="--agent" type="string">
  Filter to models that have a mapping for this agent
  
  Values: `claude`, `opencode`, `codex`, `cursor`
</ParamField>

### update

Force-fetch the latest registry from GitHub.

```bash
harness models update
```

Updates the cached model registry at `~/.harness/models.toml`.

### resolve

Resolve a model name for a specific agent.

```bash
harness models resolve <name> --agent <agent>
```

**Arguments:**

<ParamField path="name" type="string" required>
  The model name to resolve (e.g., "sonnet", "opus", "gpt-5-codex")
</ParamField>

**Options:**

<ParamField path="--agent" type="string" required>
  The agent to resolve for: `claude`, `opencode`, `codex`, `cursor`
</ParamField>

### path

Print the path to the cached registry file.

```bash
harness models path
```

**Output:** Path to `~/.harness/models.toml`

## Model registry hierarchy

Harness resolves model names through multiple registry sources:

1. **Project config** - `[models]` section in `harness.toml`
2. **Cached registry** - `~/.harness/models.toml` (fetched from GitHub)
3. **Built-in registry** - Embedded fallback models
4. **Passthrough** - Use the raw name as-is

### Example workflow

1. Use a model alias:

```bash
harness run --model sonnet --prompt "test"
```

2. Harness resolves `sonnet` → `claude-sonnet-4-20250514` for the current agent

3. The resolved model ID is passed to the agent

### Custom model mappings

Add custom models in `harness.toml`:

```toml
[models.my-sonnet]
description = "My custom Sonnet mapping"
provider = "anthropic"
claude = "claude-sonnet-4-20250514"
opencode = "claude-sonnet-4-20250514"
codex = "gpt-5-codex"
```

Now you can use:

```bash
harness run --model my-sonnet --prompt "test"
```

### Override registry models

Project config models take precedence over the cached registry:

```toml
# In harness.toml
[models.sonnet]
claude = "claude-sonnet-3-5-20241022"  # Override to use older Sonnet
```

## Use cases

### Check available models before running

```bash
harness models list --agent opencode
harness run --agent opencode --model sonnet --prompt "test"
```

### Script model selection

```bash
#!/bin/bash
# Get all models for claude agent
models=$(harness models list --agent claude --json)
echo "$models" | jq -r '.[].name'
```

### Verify model resolution

```bash
# See what model ID will actually be used
harness models resolve opus --agent claude
```

### Keep registry up to date

Periodically update the registry to get new models:

```bash
# Add to cron or CI
harness models update
```

### Debug model resolution issues

```bash
# Check what's in the registry
harness models list --json | jq '.[] | select(.name == "sonnet")'

# See where registry is stored
harness models path

# Check resolution for each agent
harness models resolve sonnet --agent claude
harness models resolve sonnet --agent opencode
harness models resolve sonnet --agent codex
```
